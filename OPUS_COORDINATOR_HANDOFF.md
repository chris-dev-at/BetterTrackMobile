## 11. Coordinator judgment — how to think (the layer above the rules)

These heuristics were each bought with a real incident on 2026-07-07. Internalize them; they are the difference between running this operation and merely following it.

1. **Builders' reports contain confident errors — verify surprising claims independently.** A builder reported "the API rejects the app's token on 0 of 104 routes" and repeated it to the owner; it was wrong. The catch: reading the platform middleware source (`apps/api/src/http/middleware/bearerAuth.ts`) and probing production with a garbage bearer (`401 API_KEY_INVALID` on `/portfolios` = bearer middleware live). Rule: any claim that changes the critical path gets an independent probe before it reaches Christian or a brief. When you correct the record, also correct the agent that holds the stale story, and fix TODO.md.
2. **Disk is truth; agent narration is not.** A dying builder narrated finished work that wasn't on disk; another's edits survived when I assumed they hadn't. On any resume/recovery: `stat` the actual files and say in the resume message exactly what exists — never let an agent trust its own memory of what it wrote. Practical: `find -newermt` proved flaky on this Mac — use `stat -f "%Sm" <file>` or `ls -lat` on known paths instead.
3. **Liveness signals, in reliability order**: (a) failure/completion task-notifications DO arrive (limit deaths notified every time — silence usually means alive); (b) fresh files in the builder's scratchpad (screenshots during device phases) are the best heartbeat; (c) SendMessage result text tells you if the process is alive ("queued for next tool round") vs dead ("resumed from transcript"). ~25–40 min of total disk silence without a notification ⇒ nudge with a status-check message that includes a bias-to-wrap-up instruction and the mandatory exit checklist. Nudged agents usually finish fine.
4. **One driver per tree and per device — batching, not parallelism, is the speed lever.** Two builders on one Gradle module or one phone will trample each other (it nearly happened; a stale agent's watchers had to be explicitly stood down). To go faster: convert the active builder into a continuous multi-step run with per-step checkpoints, and trim verification to each step's done-when. Never trim: compile-per-step, test hygiene, honest reporting.
5. **Resume beats redispatch while the model matches** (a resumed builder keeps hours of context); **after the model switch, fresh-dispatch from checkpoints** (TODO.md + disk), because resumes run on the original model. Keep briefs self-contained for exactly this reason.
6. **Never trust "cleaned up" — verify via API GET** (test transactions = 0, test portfolio archived). The owner's production account is sacred; the ` [bt:<uuid>]` note markers make test rows findable.
7. **Owner communication is a product surface.** Lead with the outcome; send the 2–4 best screenshots via SendUserFile at flagship moments (he loves seeing real screens); be exact about what is verified vs pending; surface HIS action items (platform gaps, scopes, second test account) early and repeatedly but never block on them. When an event lands mid-turn, the final chat message still has to carry everything that matters.
8. **Design QA works only when someone actually looks.** Builders must inspect their own screenshots (they catch wrong colors and misaligned digits); the coordinator spot-checks the money screens on top — that combination caught a logged-in screen showing logged-out copy and a mixed axis-label format. "It compiled" is not "it looks right".
9. **Platform-gap protocol**: openapi.json's route-security metadata lies (docs bug) — the middleware source + live probes are truth. When an endpoint is missing: build what exists, stub the rest behind a flag, note it in TODO.md, tell Christian in chat with exact endpoint names. Never invent request fields or endpoints; never re-implement server math client-side.
10. **Smallest reasonable call, noted, beats a blocking question.** Ask Christian only for owner-only things (credentials, platform/production changes, scope pivots, spending). Everything else: decide, write the decision down (TODO note or report), move on — he reads the trail later.
11. **Usage windows die mid-run — design for it.** Every brief carries "leave a compiling state and report immediately"; every multi-step run checkpoints at step boundaries; the heartbeat cron (§7) revives the loop ≤20 min after a reset and the first recovery action is always: read disk truth, then resume/dispatch. Losing a window costs an hour; losing state is a choice — never make it.

**Final word**: the standard set so far is high — exact-digit verification, honest reporting, screenshot-proofed design, clean test hygiene, the app always left working. Keep that bar. Christian's trust in this process is the product of it.
